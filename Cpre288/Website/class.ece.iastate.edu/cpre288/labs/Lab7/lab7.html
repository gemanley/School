Distance Determination

SONAR (SOund Navigation And Ranging) uses ultrasonics to determine the presence and distance of objects.  SONAR is a term that is valid for both air and water.  The frequency of the pulses emitted are different for these two applications.  We will be implementing a primitive SONAR in this lab. The Ping))) sensor from Parallax emits 40 KHz pulses.  The idea behind distance estimation is that objects reflect sound, and that the time between emitting a sound pulse and receiving an echo can be translated into distance given the speed of sound through a particular medium.  In our case the medium is air.  Air temperature does affect the speed of sound, but for our purposes the speed of sound will be constant and considered to be 340 m/s or 1130 ft/sec.

The composition and shape of objects affects the amount of sound that gets reflected back to the sensor.  A soft material like fabric will absorb more sound than a hard material like steel.  It is possible to angle a box so that sound waves will reflect away from the sensor, so the box may "disappear" like the B2 or F117 US Air Force airplanes.

The Ping))) sensor has been mounted onto the front of the VORTEX.  The Ping))) sensor will only report one echo for any given pulse.  This one echo is the first received to meet the minimum threshold used to discriminate between noise and a proper echo.  The pulse does spread as it travels away, so the first reflection may come from an object not be directly in front of the sensor.  So, clutter does affect the usefulness of the sensor.

The Ping))) sensor documentation provides the authoritative description of how it operates, but a few points are worth emphasizing here.

  1. The Ping))) sensor uses a single signal pin.  This pin is attached to the Input Capture pin of Timer 1 on Port D.

  2. The Ping))) sensor requires that it be "triggered" in order to enable pulsing.  This means that you need to set pin 4 on port D to output and then set the pin high and then low.  The documentation discusses how long the trigger pulse should be.

  3. When the Ping))) sensor pulses a green LED lights up.  It is located between the two cones.

  4. There will be a 750 uSec holdoff between trigger and receiving a signal from the sensor.  It is in this time, that the data direction on pin 4 must be transitioned from output to input.

  5. The signal received from the sensor will rise and then fall.  The distance between these two edges is directly proportional to the round-trip distance between the sensor and the object.  Input capture when properly configured will identify the edges of the signal and set ICR1 to the current counter value when the events/edges are detected.  The difference in counter values represents the signal's pulse width as time.  The actual amount of time can be determined when the pre-scaler and the system clock frequency are used to compute the amount of time each counter value represents.

  6. Initial experiments have shown the practical range of the mounted Ping))) on the floor of the lab to be roughly 5cm to 275cm.  Reflections from the floor are the likely cause for the maximum range to be limited.

Note: ATmega 128 interrupts have limited priority. If another interrupt occurs while an interrupt is being processed, the second interrupt will likely be ignored. So, keep your execution in the ISRs quick.
Alternatively, you may disable/enable all interrupts by clearing or setting the global interrupt flag.
Clear the global interrupt flag to disable all interrupts by calling cli();
Set the global interrupt flag to reenable interrupts by calling sei();

Part II. Determine the Sensor Signal's Width

Review the User Guide for using input capture and overflow interrupts on Timer 1 starting on page 133. You can look at the chart at the bottom of this page for more vectors to use in your ISR macro.

Read the PING))) documentation to get an understanding of what should happen on Pin4 of PORTD/PIND. Using input capture on Timer 1, determine the width of the Ping))) pulses in terms of the difference between the rising edge and falling edge. Display this delta on the LCD. Note that the ICR1 register is tied to Pin4 of PORTD, and that the value of the Timer/Counter 1 will be stored into ICR1 when an edge is detected (assuming you've setup Timer 1 to be used for input capture).

Using an oscilloscope, display your trigger pulse and the sensor's signal as you vary the distance between an object and the sensor.  You will be asked to show the scope traces as you demonstrate your program.  The trigger pulse is much narrower than the signal pulse, so the time scale may need to be adjusted when trying to display the pulses.  Keep in mind that Vcc is 5V and the signals will not exceed that voltage.

A couple of tutorial documents (XYZs_of_Oscilloscopes.pdf and FuncGen_Oscope_Tutorial_Revised H.pdf) have been located for those unfamiliar with the oscilloscope.  Be aware that these documents are more general than what is needed for our purposes.  The first document is not specifically for the equipment in our lab, but the concepts and techniques discussed are still relevant.

Demo the program execution on the board to your TA.

Part III. Continuous Distance Determination

Now that you have mastered pulse width calculations. Perform a distance estimation every 200 - 500 mSec and display the pulse width in time counts, in milliseconds (0 is okay for pulse widths below 1msec), and a distance calculation in centimeters. Try to be accurate with your distance calculations, you will be using them for navigation later on.

The timer is running continuously, so the counter values may span across the 2^16-1 boundary when the signal pulse is being measured. When this happens an overflow event occurs. Enable the overflow interrupt and provide a count of overflows that occur on the 2nd line of the LCD. Remember this second ISR must be very efficient. You are welcome to use the overflow event to assist with width calculation. Remember, negative widths don't make sense.

Demo the program execution on the board to your TA.

Suggestion:  All these multi-digit values need to be translated into strings before displaying them.  You might find the lprintf useful.  This function works in much the same way as printf, except that is will clear and then print to the LCD screen.
